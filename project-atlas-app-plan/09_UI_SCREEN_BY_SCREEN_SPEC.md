# 09 — UI Screen-by-Screen Specification

This document describes the screens Project Atlas must have and how they should feel.

It is written for the builder-agent that will implement the app. It should be used as a product and UX contract, not as code.

## 1. Global UI identity

Project Atlas should feel like a dark, premium, visual operating system for understanding projects.

The user should feel like they are exploring a world map of their project.

### Visual keywords

- Dark.
- Spatial.
- Calm.
- Clear.
- Premium.
- Canvas-first.
- High detail on demand.
- Smooth.
- Non-overwhelming.

### Avoid

- Light default admin-dashboard look.
- Raw JSON-first interfaces.
- Giant file tree as the main experience.
- Static documentation pages as the primary UX.
- Too many colors competing for attention.
- Technical labels as the main visible names.

## 2. Global layout pattern

Inside a project, use this structure:

```text
+--------------------------------------------------------------------------------+
| Top Bar: project switcher | lens switcher | search | import | health | diff     |
+----------------------+---------------------------------------------------------+
| Left Sidebar         | Main World Map Canvas                                  |
| - Lenses             | - zoomable systems/features/flows/items                |
| - Flows              | - minimap                                              |
| - Docs               | - filter controls                                      |
| - Warnings           | - canvas legend                                        |
+----------------------+---------------------------------------------------------+
| Optional bottom inspector: trace replay, diff details, validation issues         |
+--------------------------------------------------------------------------------+
                                      +------------------------------------------+
                                      | Right Drawer: selected item details       |
                                      +------------------------------------------+
```

The canvas should always feel like the center of the product.

## 3. Screen: Project Library

### Purpose

Let the user choose or import a project atlas.

### Main user question

```text
Which project do I want to understand or review?
```

### Must show

- App title: Project Atlas.
- Short tagline: Visual maps for understanding code projects.
- Project cards.
- Import button.
- Demo project button.
- Empty state if no projects exist.

### Project card content

Each card should show:

- Project name.
- Summary.
- Source/repository/path.
- Last imported.
- Counts: systems, flows, docs, scripts, MCP tools, hooks, warnings.
- Health indicator.
- Open action.

### Visual style

Dark page with large elegant cards. Project cards should look like portals into project worlds.

### Empty state copy

```text
No project atlases imported yet.

Ask an agent to create .project-atlas/atlas.bundle.json inside a target project, then import it here.
```

### Acceptance

A first-time user knows exactly how to begin.

## 4. Screen: Import Project

### Purpose

Let the user import an atlas bundle.

### Main user question

```text
How do I add a project to Project Atlas?
```

### Must support first

- Import `atlas.bundle.json`.
- Use demo bundle.

### Later support

- Import `.project-atlas/` folder.
- Import zip.
- Import from local/desktop helper.
- Import from remote source.

### Must show

- Drop zone.
- File picker.
- Explanation of required file.
- Link/copyable prompt for target-project agents.

### Import explanation

```text
Project Atlas does not need to read your whole codebase directly.
It reads a structured atlas bundle created by an agent or scanner inside your target project.
```

### Acceptance

User can import the example bundle and reach the validation screen.

## 5. Screen: Import Validation

### Purpose

Explain whether the imported atlas bundle is usable.

### Main user question

```text
Can this project map be trusted enough to open?
```

### Must show

- Valid/invalid status.
- Schema version.
- Generated time.
- Generated by.
- Project name.
- Counts:
  - nodes
  - edges
  - flows
  - artifacts
  - documents
  - traces
  - warnings
- Blocking errors.
- Non-blocking warnings.
- Suggested fixes.
- Open project action if safe.

### Error card format

```text
Problem: Edge points to a missing node.
Where: edge_form_calls_create_project_api
Why it matters: The relationship cannot be shown on the map.
Agent fix: Add the missing node or update the edge target.
```

### Visual style

Validation should feel like a system scan result: precise, calm, helpful.

### Acceptance

The user can understand whether a failed import is an app problem or an atlas data problem.

## 6. Screen: Project Home

### Purpose

Give a high-level starting point before entering specific lenses.

### Main user question

```text
What is this project, and what should I look at first?
```

### Must show

- Project name.
- Project summary.
- Source repo/path.
- Last updated.
- Atlas health summary.
- Main counts.
- Entry cards:
  - Open World Map.
  - Feature Flows.
  - Data Map.
  - Agent/MCP/Hooks.
  - Docs and AGENTS.md.
  - Folder Structure.
  - Scripts and Workflows.
  - Runtime Traces.
  - Change Impact.
  - Snapshot Diff.
- Warnings/needs-review section.

### Visual style

Dashboard-like, but still spatial and polished. Entry cards should feel like doorways into lenses.

### Acceptance

The user can decide where to go without understanding code terms.

## 7. Screen: Overview World Map

### Purpose

Show the whole project as a canvas-first system map.

### Main user question

```text
What exists in this project, and how does it connect?
```

### Default visible items

- Project root.
- Major systems/domains.
- Important external services.
- Important data stores.
- Important automation/agent/deployment areas.
- Top-level relationships.

### Must not show by default

- Every file.
- Every tiny function.
- Every package dependency.
- Every low-level edge.

### Canvas layout metaphor

Use a world-map layout:

```text
Left: user/front-facing surfaces
Center-left: frontend
Center: backend/business logic
Center-right: data
Right: external services
Top/right: agent tooling and automation
Bottom/right: deployment and runtime
```

### Canvas interactions

- Pan around project world.
- Zoom in/out.
- Click node.
- Hover to preview.
- Focus on node.
- Collapse/expand system.
- Highlight connections.
- Open detail drawer.
- Switch lens without leaving context.

### Visual states

- Selected node glows subtly.
- Connected nodes brighten.
- Unrelated nodes dim.
- Warnings show small badges.
- Low-confidence items show distinct badge.
- Runtime-observed items show runtime marker.

### Acceptance

The user can understand the main systems in under one minute.

## 8. Screen: Focus Mode

### Purpose

Let the user inspect one item without losing context.

### Main user question

```text
What directly connects to this thing?
```

### Trigger

- Select node and click Focus.
- Keyboard shortcut.
- Command palette action.

### Must show

- Focus node in center.
- Direct upstream nodes.
- Direct downstream nodes.
- Related flows.
- Related docs/artifacts.
- Dimmed surrounding context.

### Example

Focusing on `Project Service` shows:

- Create Project API upstream.
- Users Table read.
- Projects Table write.
- Activity Logs Table write.
- Email Service use.
- Create Project flow.
- Project Service file.

### Acceptance

The user can understand one part without being overwhelmed by the whole graph.

## 9. Screen: Flow List

### Purpose

Show all known flows/pipelines/processes.

### Main user question

```text
What things can happen in this project?
```

### Must show

- Flow name.
- Flow type.
- Trigger.
- Area.
- Step count.
- Confidence.
- Runtime trace availability.
- Warning marker.

### Filters

- Flow type.
- Area.
- Confidence.
- Has runtime trace.
- Has warnings.

### Acceptance

The user can find key flows like Create Project, Stripe Webhook, CI Build, and Agent Creates Pull Request.

## 10. Screen: Flow Detail

### Purpose

Show a flow step by step.

### Main user question

```text
What happens from start to finish?
```

### Required modes

1. Timeline.
2. Canvas.
3. Sequence.

### Timeline mode

Best default for non-coders.

Each step card should show:

- Step number.
- Step name.
- Simple summary.
- Inputs.
- Outputs.
- Reads/writes.
- Supporting files/docs/scripts/tools.
- Error paths.
- Confidence/evidence.

### Canvas mode

Shows steps spatially, with supporting files/tools/docs around each step.

### Sequence mode

Shows actor-to-actor movement:

```text
User -> Frontend Form -> API -> Auth -> Service -> Database -> Email Service
```

### Step click behavior

Clicking a step opens detail drawer with all related artifacts and relationships.

### Acceptance

A non-coder can explain the Create Project flow after reading the timeline.

## 11. Screen: Data Lens

### Purpose

Show where data lives and how data moves.

### Main user question

```text

```

### Must show

- Databases.
- Tables/collections.
- Buckets/queues/caches if present.
- Read edges.
- Write edges.
- Sensitive data markers.
- Flow overlays.

### Interaction

Selecting a table should show:

- What it stores.
- Who reads it.
- Who writes it.
- Related flows.
- Related schema/migration files.
- Sensitive marker.

### Acceptance

The user can see that Project Service reads Users and writes Projects/Activity Logs.

## 12. Screen: Dependency Lens

### Purpose

Show what depends on what.

### Main user question

```text

```

### Must show

- Upstream dependencies.
- Downstream dependents.
- Internal service dependencies.
- Flow dependencies.
- External service dependencies.
- Script/workflow dependencies.

### Required interaction

Click any node and choose:

- Show upstream.
- Show downstream.
- Show both.
- Show impact.

### Acceptance

The user can inspect dependencies without reading import statements.

## 13. Screen: Agent / MCP / Hooks Lens

### Purpose

Make hidden automation and agent tooling visible.

### Main user question

```text
What agents, tools, MCPs, hooks, and automations exist in this project?
```

### Must show

- Agent Tooling system.
- Agents.
- AGENTS.md files.
- MCP servers.
- MCP tools.
- MCP config files.
- Hooks by category.
- Webhooks.
- CI/workflow triggers.
- Agent update instructions.

### Required drill-down path

```text
Agent Tooling
  -> GitHub MCP Server
    -> Read Repository Tool
    -> Create Pull Request Tool
  -> Root AGENTS.md
  -> Implementation Agent
```

### Hook categories

The UI must distinguish:

- Framework hooks.
- Integration webhooks.
- Git hooks.
- Package lifecycle hooks.
- Agent hooks.
- Automation hooks.

### Acceptance

The user can click an MCP server and see its tools, then click a tool and see what flow uses it.

## 25. Global empty states

Empty states should never be dead ends.

Every empty state should explain:

1. What is missing.
2. Whether this means the project lacks it or the atlas simply has not mapped it.
3. What an agent should do next.

Example:

```text
No runtime traces are mapped yet.

The project may still work normally. This only means Project Atlas has not received observed trace data.
Ask an agent to capture logs or traces for important flows and add them to .project-atlas/atlas.traces.json.
```

## 26. Global visual rules

### Typography

- Use readable sans-serif for UI.
- Use monospace only for paths, commands, IDs, and code-like values.

### Color

- Use dark background.
- Use restrained accents.
- Use clear badge colors.
- Do not rely on color alone.

### Spacing

- Canvas can be spacious.
- Drawers can be dense but organized.
- Avoid cramped cards.

### Motion

Use subtle transitions:

- Opening drawer.
- Focusing node.
- Expanding cluster.
- Switching lens.
- Zooming canvas.

Motion should help orientation, not distract.

## 27. First-run guided experience

When the user opens the demo project for the first time, the app should guide them through:

1. This is the project world map.
2. These are the main systems.
3. Try clicking Create Project flow.
4. Try opening AGENTS.md.
5. Try searching for MCP.
6. Try showing impact for Project Service.

This can be a lightweight overlay or checklist.

## 28. Final UX acceptance standard

The app is successful when the user can say:

```text
I can see the project like a map.
I can zoom out to understand the whole system.
I can zoom in to inspect flows, files, docs, scripts, MCPs, hooks, and data.
I can click AGENTS.md and read it in the app.
I can find what I want without knowing code.
I can ask an agent to update the atlas when the project changes.
```
